Composition vs. Inheritance

Matan Keidar Pipl

In the beginning...

  • Inheritance is a powerful mechanism in OOP
  • Very popular since it was introduced
  • However, seems to be anti-pattern in these days
    • Some modern languages do not allow inheritance
    • They allow composition instead
  • Our goal is to understand why this is happening ü§î

Inheritance

Inheritance

  • A child class inherits from a parent class
  • The child class acquires all behvaiors from the parent
  • Inheritance creates class hierarchy

Composition

Composition

  • Composition is in contrast to inheritance
  • Creates new types by combining objects of other types
    • Rather than inheriting from a base class
  • In other words: composition contains instances of another classes

Delegates


							trait Ord[T] {
							  def (x: T) compareTo (y: T): Int
							  def (x: T) < (y: T) = x.compareTo(y) < 0
							}

							delegate IntOrd for Ord[Int] {
							  def (x: Int) compareTo (y: Int) = 
							    if (x < y) -1 else if (x > y) 1 else 0
							}

							def maximum[T](xs: List[T]) given Ord[T] = 
							  xs.reduce((x,y) => if (x < y) y else x)
  • Replaces implicit as a modifier
  • Express intent instead of mechanism

Delegates under the hood

Without parame[T]ers:


							delegate IntOrd for Ord[Int] { ... }
maps to: 

							implicit object IntOrd extends Ord[Int] { ... }

Delegates under the hood

With parame[T]ers:


						    delegate ListOrd[T] for Ord[List[T]] given (ord: Ord[T]) { ... }
maps to: 

							class ListOrd[T](implicit ord: Ord[T]) extends Ord[List[T]] { 
							  ... 
							}
							
							final implicit def ListOrd[T](implicit ord: Ord[T]): ListOrd[T] = 
							  new ListOrd[T]

Delegates


							trait SemiGroup[T] {
							  def (x: T) combine (y: T): T
							}

							trait Monoid[T] extends SemiGroup[T] {
							  def unit: T
							}

							delegate for Monoid[String] {
							  def (x: String) combine (y: String) = x concat y
							  def unit= ""
							}

							def sum[T: Monoid](xs: List[T]): T =
							  xs.foldLeft( the[Monoid[T]].unit )(_ combine _)
  • Works out of the box with extension methods
  • Provides a nice syntax for typeclasses

Delegates

Tame Implicit Conversions


							implicit def strToToken(str: String): Token = 
							  new Keyword(str)
What's wrong with it?
  • Very easy to write vs. how dangerous it is
  • implicit as a modifier will go away
    • and with it this kind of conversion

Delegates

Only way to express implicit conversions
is as a delegate to Conversion class: 

							delegate for Conversion[String, Token] {
							  def apply(str: String): Token = new KeyWord(str)
							}
Or using an alias delegate: 

							delegate for Conversion[String, Token] =
							  new KeyWord(_)

Why Delegates?

  • implicits are Scala's most distinguished feature.
  • But they are also the most controversial one.

  • Delegates are simpler and safer alternative:
    • Emphasize intent over mechanism
    • Make idea of term inference more accessible
    • Discourage abuses

Opaque Types

Provide type abstraction without any overhead

Opaque Types


							object Logarithms {
							  opaque type Logarithm = Double
								
							  object Logarithm {
							    // These are the ways to lift to the logarithm type
							    def apply(d: Double): Logarithm = math.log(d)
							    def safe(d: Double): Option[Logarithm] =
							      if (d > 0.0) Some(math.log(d)) else None
							  }
							  
							  delegate LogarithmOps {
							    def (x: Logarithm) toDouble: Double = math.exp(x)
							    def (x: Logarithm) + (y: Logarithm): Logarithm = Logarithm(math.exp(x) + math.exp(y))
							    def (x: Logarithm) * (y: Logarithm): Logarithm = Logarithm(x + y)
							  }
							}

Opaque Types

  • Introduces Logarithm as a new type
  • Implemented as a Double but different from it
  • Logarithm is treated as abstract type: 
      
									val l1 = Logarithm(1.0)
									val l2 = Logarithm(2.0)
									val l3 = l1 * l2
									val l4 = l1 + l2
									val d: Double = l1       // error
									val l2: Logarithm = 1.0 // error

Match Types


							type Elem[X] = X match {
							  case String      => Char
							  case Array[T]    => T
							  case Iterable[T] => T
							}
A match type reduces to one of a number of right hand sides, depending on a scrutinee type.

Match Types


							enum Tuple {
							  case Empty
							  case Pair[H, T]
							}

							import Tuple._

							type Concat[+Xs <: Tuple, +Ys <: Tuple] <: Tuple = 
							  Xs match {
							    case Empty => Ys
							    case Pair[x, xs] => Pair[x, Concat[xs, Ys]]
							  }
  • Match types can be recursive.
  • More straightforward than using implicits
    • (c.f. HLists)
  • Work in type checking is still in progress
